每次函数调用在栈上分配一个栈帧，包含局部变量、返回地址（LR/PC）和上一帧的 FP。 栈展开就是沿着 FP 链或 DWARF unwind table 逐帧回溯这些返回地址。

unw_step() 返回正值时表示成功展开到上一帧，返回 0 表示已到达栈顶。 所有操作仅读取进程地址空间中已映射的数据，不调 malloc/printf。

信号 handler 被中断的代码可能正持有锁（如 malloc 内部锁）。 如果 handler 中再次尝试获取同一把锁 → 死锁。 本库通过只用左侧的操作来彻底规避。

5. Core dump 与堆栈文本的互补关系

sequenceDiagram participant App as 应用程序 participant Kernel as 内核 participant Handler as crash_signal_handler participant Lib as swp_stack_trace participant Disk as 磁盘 App->>Kernel: 触发 SIGSEGV Kernel->>Handler: 调用注册的 handler Handler->>Lib: print_stacktrace(STDERR_FILENO) Lib->>Disk: 写入 stderr → 终端/日志文件 Note over Lib,Disk: ① 先打印: 保留原始崩溃现场的调用栈 Handler->>Handler: signal(sig, SIG_DFL) Handler->>Kernel: raise(sig) Kernel-->>Kernel: 默认行为: 终止 + core dump Kernel->>Disk: 写入 core 文件 Note over Kernel,Disk: ② 再 core: 保留完整内存快照 Note over App,Disk: 文本堆栈 (原始现场) + core dump (内存快照) = 完整事后分析

先打印堆栈文本，再 re-raise 生成 core dump。两者互补： 文本记录崩溃时的调用链，core dump 提供完整内存用于 gdb 深入分析。
